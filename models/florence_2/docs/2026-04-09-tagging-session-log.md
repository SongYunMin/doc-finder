## 2026-04-09 기준 전제와 결정

### 공통 전제

- CMS는 source of truth로 유지한다.
- 검색 시스템은 문항 전체 JSON이 아니라 참조 키와 태그 인덱스를 관리한다.
- 1차 검색은 문항 본문 검색이 아니라 이미지 태그 검색이다.
- 파일명은 기본적으로 `unitId_dataId_imageOrder`를 따르되, `imageOrder`가 없으면 `1`로 본다.
- 대량 이미지 원본은 프로젝트 내부에 넣지 않고 외부 경로를 참조하는 것이 맞다.
- 데이터셋에는 일반 사물 일러스트와 문제 도식/기하 이미지가 섞여 있을 가능성이 높다.

### 주요 결정

- 검색 API는 `unit_id`, `data_id`, `image_id`, `cms_ref` 중심으로 가고 문항 JSON 조인은 후속 단계로 분리한다.
- 기준 식별자는 `unit_id + data_id + image_id`로 고정한다.
- `10709_13048.svg`처럼 `imageOrder`가 없는 파일도 허용하고 `image_id=1`로 처리한다.
- 현재 MVP 구현을 그대로 30만 장에 바로 투입하는 것은 위험하다고 판단했다.

## 2026-04-09 작업 요약

### 1. `static` 태거 스모크 테스트

- `static-tags.json`과 샘플 이미지로 `ingest -> search`를 검증했다.
- 문제는 세 가지였다.
  - `static-tags.json`를 이미지 폴더 아래 두면 스캔에 같이 잡혀 `invalid_filename` reject 발생
  - JSON 키와 실제 파일 확장자가 다르면 `No static tags configured ...` 발생
  - `DOC_FINDER_DATABASE_URL`이 없으면 `InMemory` 저장소로 떨어져 ingest/search 결과가 이어지지 않음
- 조치:
  - `static-tags.json` 분리
  - JSON 키와 실제 파일명 일치
  - DB URL을 같은 셸에서 명시적으로 로드
- 결과적으로 `static` 모드 스모크 테스트는 성립했다.

### 2. Postgres 연결 문제

- `5431` 포트 접속 실패 원인은 Docker 매핑 오류였다.
- 컨테이너를 `5431 -> 5432`로 다시 띄운 뒤 `postgresql://postgres:postgres@localhost:5431/doc_finder` 접속이 정상화됐다.

### 3. 검색 결과 이상 동작 확인

- `search` 결과가 비어 있던 이유 중 하나는 실제 `image_index` row가 0건이었기 때문이다.
- 또 다른 원인은 exact hit가 없어도 semantic fallback 후보가 그대로 남는 현재 검색 그래프 동작이었다.
- 따라서 `matched_tags=[]` 결과는 “태그 적중”이 아니라 semantic-only 후보로 해석해야 한다.
- 이 단계에서는 검색 랭킹보다 태그 품질을 먼저 보는 것이 맞다고 정리했다.

### 4. Florence-2 통합

- `static/http` 구조를 유지한 채 `florence2` provider를 추가했다.
- 초기에는 `transformers` 호환성 이슈가 있었고, `torch_dtype` 정리와 `attn_implementation="eager"`로 정리했다.
- `einops`, `timm` 누락 문제도 추가 설치로 해결했다.
- SVG는 `cairosvg`로 `SVG -> PNG bytes -> PIL RGB` 변환 경로를 넣어 태깅 가능하게 만들었다.

### 5. Florence-2 품질 한계 확인

- 일반 사물 이미지가 아닌 문제 도식/기하 이미지에서 Florence-2가 매우 부정확했다.
- 대표 오탐은 `person`, `wheelchair`, `grass`, `black`, `white`였다.
- 문장형 캡션을 후처리로 잘게 줄여도 도식 이미지 품질은 크게 좋아지지 않았다.
- 결론적으로 Florence-2를 전체 데이터셋 주력 태거로 바로 가는 방향은 재검토가 필요하다고 판단했다.

### 6. WD tagger 비교 판단

- 수학/기하 도식 이미지가 주요 샘플이라면 `WD tagger`도 근본 해결책이 아닐 가능성이 높다고 봤다.
- 이유는 WD tagger가 Danbooru/애니 일러스트 태그 편향이 강하고, 현재 필요한 태그가 문제 구조 태그이기 때문이다.
- 따라서 데이터셋이 일반 사물 / 문제 도식으로 나뉜다면 태깅 전략도 분기하는 쪽이 더 현실적이다.

- 외부 유료 API는 샘플 평가용 후보로만 보고, 장기 운영 경로는 비용 구조를 먼저 확인해야 한다.

### 구조/학습 방향 추가 정리

#### 1. Florence 관련 구조 정리

- Florence 런타임 코드는 `src/doc_finder/models/florence_2` 아래에 두고, 루트 `models/florence_2`는 문서와 실험 기록만 보관하는 쪽으로 다시 정리했다.
- 중간에 루트 `models`를 실제 Python 패키지처럼 쓰는 구조도 검토했지만, 이 방식은 앱 런타임 코드와 실험 자산의 경계를 흐리기 때문에 장기적으로 불리하다고 판단했다.
- 따라서 기준은 아래처럼 고정했다.
  - 런타임 코드: `src/doc_finder/models/*`
  - 실험/문서: `models/*`
  - 공용 입력 이미지: 루트 `images/`
  - 정적 태그 fixture: 루트 `static-tags.json`
- 이 판단의 핵심은 “모델이 늘어날수록 앱 코드와 실험 자산은 더 엄격히 분리해야 한다”는 점이다.

#### 2. Florence-2 fine-tuning 준비 방향 고정

- Hugging Face의 Florence-2 fine-tuning 글을 그대로 DocVQA 작업으로 따라가지 않고, 현재 프로젝트에 맞는 `GeoTag` 작업으로 재해석하는 쪽으로 방향을 고정했다.
- 여기서 `GeoTag`는 자유문장 생성이 아니라 아래 같은 표준 태그 문자열 생성으로 정의했다.

```text
triangle; point_a; point_b; angle_label; parallel
```

- fine-tuning 준비물은 루트 `models/finetune_florence_2` 아래에 정리했다.
- 현재 기준으로 필요한 단계 수는 총 8단계로 정리했다.
  - 작업 정의
  - taxonomy 설계
  - 골드 라벨 구축
  - 데이터 분할
  - zero-shot baseline 측정
  - 학습 포맷 준비
  - 소규모 smoke fine-tuning
  - 본 학습과 평가
- 특히 “일단 학습부터 돌린다”는 접근은 피하고, 최소 1~5단계가 끝나기 전에는 본 학습에 들어가지 않는 쪽이 맞다고 정리했다.

#### 3. fine-tuning 1단계에 대한 추가 판단

- 1단계는 학습 단계가 아니라 “정답 형식 계약”을 고정하는 단계로 봤다.
- 즉, 모델이 최종적으로 어떤 문자열을 생성하면 정답으로 볼지 먼저 정해야 한다.
- 여기서 중요한 판단은 세 가지다.
  - 현재 모델 출력이나 `keyword_tags`를 학습 정답으로 그대로 쓰지 않는다.
  - 동의어를 허용하지 말고 대표 표기 하나만 허용한다.
  - 자유문장 대신 세미콜론 구분 표준 태그 문자열로 고정한다.
- 이 단계가 불안정하면 이후 taxonomy, 라벨링, 평가가 모두 흔들리므로, 실제 학습보다 우선순위가 높다고 봤다.

#### 4. 재부팅 이후 Postgres 연결 실패 원인 확인

- `5431` 포트 연결 실패는 DB URL 오타가 아니라, 재부팅 후 `doc-finder-pg` 컨테이너가 자동으로 다시 올라오지 않은 것이 원인이었다.
- 실제로 `5431`을 담당하던 컨테이너는 `Exited (255)` 상태였고, restart policy가 `no`였다.
- 반면 다른 프로젝트용 Postgres만 `5432`에서 정상 동작 중이었다.
- 따라서 문제는 “앱 설정”보다 “로컬 인프라 재기동 정책”에 있었다.
- 수동으로 `doc-finder-pg`를 다시 시작한 뒤 `5431` 포트가 다시 열리는 것까지 확인했다.

#### 5. 이번 세션 기준 결론

- Florence-2 자체보다도 프로젝트 내부 구조와 실험 절차를 먼저 안정시키는 것이 맞다.
- 모델 코드와 실험 자산은 분리하고, fine-tuning은 준비 단계를 건너뛰지 않는 쪽으로 간다.
- 로컬 DB 연결 문제도 애플리케이션 코드보다 컨테이너 생명주기와 재시작 정책을 먼저 확인하는 것이 효율적이라는 점을 다시 확인했다.

#### 6. PEFT/LoRA fine-tuning 방향 재정리

- 기존 `models/florence_2_tuning` 초안은 full fine-tuning 성격이 강했지만, 현재 목표에는 `PEFT/LoRA` 중심으로 다시 정리하는 쪽이 더 맞다고 판단했다.
- 이유는 두 가지다.
  - 현재 문제는 단순 모델 크기보다 데이터 도메인 적합성과 실험 반복 속도가 더 중요하다.
  - Florence-2 전체를 직접 학습하는 방식은 메모리와 저장 단위가 커서 실험 회전이 느려진다.
- 다만 여기서도 “무조건 adapter-only”로 부르는 것은 부정확하다고 봤다.
- Florence-2에는 `image_projection`, `image_proj_norm`, `image_pos_embed`, `visual_temporal_embed` 같은 visual bridge 계층이 있고, 이 중 일부를 일반 trainable weight로 함께 다루는 편이 더 현실적일 수 있기 때문이다.
- 따라서 현재 기준의 핵심 판단은 아래다.
  - language model 쪽은 LoRA로 간다.
  - vision tower는 기본 freeze를 유지한다.
  - visual bridge는 필요한 범위만 선택적으로 full-train 한다.
  - 저장 단위는 adapter 하나만이 아니라 `adapter + extra trainables + metadata` 묶음으로 보는 것이 더 정확하다.

#### 7. 학습 입력 계약에 대한 추가 판단

- 현재 목표는 `png`뿐 아니라 `svg`도 학습 입력으로 쓰는 것이다.
- 하지만 Florence-2가 벡터 SVG를 직접 이해하는 것은 아니므로, SVG는 학습 직전에 deterministic한 raster RGB 이미지로 바꿔서 넣는 것이 맞다고 정리했다.
- 여기서 더 중요한 판단은 “학습 전처리와 나중의 추론 전처리가 달라지면 안 된다”는 점이다.
- 즉, square padding이나 배경 합성 같은 전처리를 학습 쪽에만 새로 넣으면 이후 finetuned 모델을 실제로 붙일 때 입력 분포 mismatch가 생길 수 있다.
- 그래서 fine-tuning 자체보다 먼저 아래 계약을 고정해야 한다고 봤다.
  - 어떤 입력 형식을 받을지
  - SVG를 어떤 설정으로 rasterize할지
  - 모든 raster 입력을 어떤 공통 전처리로 맞출지

### 1. DB를 건드리지 않는 `tag preview` 경로를 별도로 두는 것이 맞다고 정리

- 기존 `ingest` 경로는 `sha256` 중복 체크와 `(unit_id, data_id, image_id)` upsert를 같이 쓰므로, 같은 저장소에서 모델 비교를 하면 “기존 로직에 영향 없이 비교한다”는 목표가 성립하지 않는다고 봤다.
- 그래서 비교 목적의 실행은 `ingest`에 옵션을 덧붙이는 방식보다, 저장을 전혀 하지 않는 preview 전용 CLI로 분리하는 편이 더 안전하다고 정리했다.
- 이 경로에서 유지한 기준은 아래다.
  - 디렉터리 순회 정책은 `ingest`와 같게 유지한다.
  - 파일명 검증과 이미지 메타데이터 검증은 재사용한다.
  - 태깅은 같은 provider 계약을 그대로 탄다.
  - 저장소 생성, upsert, reject 기록, embedding 계산은 하지 않는다.
- 핵심 판단은 “비교용 실행 경로와 운영 적재 경로를 섞지 않는다”는 점이다.

### 2. `tag preview` 출력은 사람 비교 중심으로 보는 것이 맞다고 정리

- preview CLI의 출력은 한 줄짜리 raw JSON보다, 사람이 바로 비교할 수 있는 줄바꿈 JSON이 더 적합하다고 정리했다.
- 여기서 중요한 판단은 두 가지였다.
  - preview 경로의 1차 목적은 기계 간 연동보다 샘플 결과를 빠르게 육안 검토하는 것이다.
  - 다만 구조 자체는 JSON을 유지해야 하므로, 포맷만 pretty JSON으로 바꾸고 스키마는 유지하는 것이 맞다.
- 즉 이 단계의 출력 설계는 “새 데이터 형식 정의”가 아니라 “비교용 가독성 최적화”로 봤다.

### 3. 모델 provider 분리 원칙을 실제 코드 구조로 확장

- 기존에 말로만 정리하던 “모델별 런타임 분리”를 실제 provider 구조까지 확장했다.
- 기준은 아래처럼 다시 고정했다.
  - 기본 Florence-2: `florence2`
  - finetuned Florence-2: `florence2-large-ft`
  - 비교용 PaliGemma 2: `paligemma2`
- 중요한 점은 “모델 id만 바꿔서 같은 태거를 억지로 돌리는 방식”보다, 모델 책임이 다르면 provider 이름과 런타임 네임스페이스도 분리하는 편이 더 안전하다는 점이다.

### 4. PaliGemma 2는 붙일 수 있지만 접근 제약이 먼저 있다는 점 확인

- `paligemma2` provider 자체는 Florence와 같은 구조로 프로젝트 안에 붙일 수 있다는 점을 확인했다.

### 5. PaliGemma 2 초기 품질은 검색용 태거 기준으로 아직 불안정
